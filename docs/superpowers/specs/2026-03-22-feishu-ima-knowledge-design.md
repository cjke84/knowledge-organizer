# 飞书知识库与腾讯 IMA 接入设计

## 背景

当前 `knowledge-organizer` 的主流程已经支持：

- 文章链接、Markdown 草稿、结构化 draft
- 重复检测
- 标签与摘要生成
- 图片下载到 `assets/`
- 输出 Obsidian 原生 Markdown

用户现在希望在此基础上新增两个目标端：

- 飞书知识库
- 腾讯 IMA 知识库

并且要求：

- 只做导入 / 检索
- 支持一次性导入
- 支持定时增量同步
- 参考飞书官方插件和 IMA 官方插件的能力模型

## 目标

1. 统一现有输入来源，避免为每个目标端重复写一套解析逻辑。
2. 将飞书和 IMA 作为独立目标适配器接入。
3. 支持三类输入：
   - 文章链接 / 网页内容
   - Markdown / 草稿文件
   - 文件夹扫描
4. 支持两种运行模式：
   - 一次性导入
   - 定时增量同步
5. 保留现有 Obsidian 输出能力，不破坏当前 skill。

## 非目标

- 不做双向同步
- 不做目标端内容编辑回写
- 不做复杂权限管理
- 不在第一版里支持所有第三方知识库平台

## 设计原则

- 统一入口，目标端解耦
- 先归一化，再输出
- 去重和幂等优先于“看起来跑过了”
- 目标端差异关在适配器里，不污染主流程
- IMA 若公开 API 细节不足，先保留可验证的导入路径，不硬猜接口

## 总体架构

### 1. Source 层

负责读取内容来源：

- URL
- Markdown 文件
- 文件夹扫描

职责：

- 抽取标题、正文、图片、来源信息
- 统一格式，输出 `ImportDraft`

### 2. Normalizer 层

负责把不同来源整理成统一中间模型。

统一字段建议：

- `title`
- `aliases`
- `source_type`
- `source_url`
- `content`
- `summary`
- `bullets`
- `tags`
- `images`
- `attachments`
- `published`
- `created`
- `updated`
- `author`
- `engagement`
- `raw_content_markdown`

### 3. Dedup 层

负责判定是否已存在可复用内容。

判定信号：

- `source_url`
- 标题
- 内容 hash
- 别名
- 文本相似度

### 4. Destination 层

输出目标分为两个适配器：

- `feishu_knowledge`
- `ima_knowledge`

每个适配器只关心：

- 如何把 `ImportDraft` 转成目标端负载
- 如何创建目标内容
- 如何更新或跳过重复内容

### 5. Sync State 层

记录同步状态，支持一次性导入和定时增量同步。

建议记录：

- `source_id`
- `source_hash`
- `content_hash`
- `destination`
- `remote_id`
- `remote_url`
- `last_synced_at`
- `status`
- `error_message`

## 飞书接入设计

飞书适配器优先对齐官方插件 / 云文档 / 知识库体系的能力模型。

第一版重点：

- 导入文章或 Markdown 到飞书知识库结构中
- 保留标题、正文、图片、标签、来源
- 支持幂等导入

实现方式：

- 将 `ImportDraft` 转成飞书可接受的文档结构
- 通过飞书官方授权和 API 创建内容
- 记录返回的 `remote_id` 和 `remote_url`

## IMA 接入设计

IMA 适配器优先对齐腾讯 IMA 官方插件/知识库的能力模型。

当前设计保留两条路径：

1. 如果官方 API / 导入接口足够明确，则直接走 API 接入。
2. 如果公开 API 不足，则先保留“官方可验证导入路径”作为适配器实现，不假设不存在的接口。

第一版目标：

- 导入文章、Markdown、文件夹内容
- 保留结构化元数据
- 支持增量同步的状态记录

## 数据流

1. 输入来源进入 Source 层
2. Source 层输出 `ImportDraft`
3. Normalizer 统一字段
4. Dedup 判断是否已有相同内容
5. Destination 选择飞书或 IMA
6. 适配器推送到目标端
7. Sync State 记录结果

## 一次性导入

一次性导入适合：

- 迁移历史文章
- 批量导入已有 Markdown
- 扫描一个文件夹后全部入库

规则：

- 先去重
- 再导入
- 成功后写入同步状态
- 部分失败不影响其他项

## 定时增量同步

增量同步适合：

- 目录持续更新
- 草稿文件频繁修改
- 新增文章链接持续进入

规则：

- 以 `source_hash` / `content_hash` 识别变化
- 对比上次同步记录
- 只同步新增或变更内容
- 对已同步且未变化的内容跳过

## 错误处理

- 认证失败
  - 记录错误
  - 中止当前目标端任务
  - 提示重新授权
- 权限不足
  - 跳过该条
  - 不影响其他条目
- 目标端冲突
  - 以幂等更新或跳过处理
- IMA 路径不明确
  - 先走可验证导入路径
  - 不在文档没确认前硬写接口
- 部分成功
  - 保留状态
  - 支持下次续跑

## 测试策略

### 单元测试

- `ImportDraft` 归一化
- 去重判定
- 同步状态写入
- 目标适配器选择

### 集成测试

- 飞书适配器 mock
- IMA 适配器 mock
- 一次性导入流程
- 增量同步流程

### 回归测试

- 同一内容重复导入不会重复写
- Markdown / 链接 / 文件夹三类来源输出一致
- 飞书 / IMA 适配器互不影响

## 交付范围

第一版只交付：

- 统一导入模型
- 飞书目标适配器
- IMA 目标适配器
- 一次性导入
- 定时增量同步
- 测试覆盖

不包含：

- 双向同步
- 复杂权限路由
- 目标端内容编辑器

## 需要验证的点

1. 飞书官方知识库 / 云文档的具体导入接口是否适合直接复用。
2. 腾讯 IMA 官方插件是否提供稳定可调用的导入入口。
3. 两个平台对图片、附件、标签、来源字段的保真程度。

## 结论

这套设计的核心不是“把 Obsidian 改成飞书/IMA”，而是把当前知识整理链路升级成一个通用导入框架。

这样以后新增第三方知识库时，只需要再补一个目标适配器，而不是重写整条链路。
